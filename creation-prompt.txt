OBJECTIF FONCTIONNEL

Le projet doit permettre :

- d’utiliser Open WebUI comme interface utilisateur
- d’interroger un corpus documentaire via Microsoft GraphRAG
- d’appeler un endpoint LLM externe compatible OpenAI
- d’intégrer GraphRAG via un service bridge FastAPI
- d’utiliser un pipeline Open WebUI pour interroger GraphRAG
- d’exposer ce pipeline comme modèle réellement consommable par Open WebUI via l’API OpenAI-compatible du service pipelines
- d’utiliser par défaut un corpus GraphRAG local sous graphrag/input, distinct des espaces documentaires d’Open WebUI tant qu’aucune synchronisation explicite n’est réellement implémentée
- d’exposer un visualiseur interactif du graphe GraphRAG, accessible par URL et exploitable depuis Open WebUI via un lien renvoyé par le pipeline
- de faire en sorte que ce visualiseur reste utile même sans index GraphRAG complet : si les artefacts `entities.parquet` / `relationships.parquet` / `graph.graphml` ne sont pas encore prêts, le bridge doit fournir un graphe documentaire de secours construit directement depuis les fichiers du corpus
- de prévoir une image de modèle Open WebUI issue de la mascotte, appliquée explicitement via `meta.profile_image_url` sur les entrées de modèles GraphRAG ou leurs overrides/aliases
- d’utiliser un SSO Keycloak en environnement Kubernetes
- d’automatiser le build, le déploiement, les tests et les vérifications de sécurité

- Open WebUI peut enrichir les modèles GraphRAG avec des overrides/aliases persistés dans sa base (`webui.db`) afin d’y associer un nom affiché, une description et une image de profil de modèle

EXTENSION STRUCTUREE : MODELES GENERALISTES ET RECHERCHE INTERNET

Le repository généré doit aussi couvrir ce besoin complémentaire de manière structurée :

1. Ajouter dans Open WebUI une famille de modèles généralistes Scaleway pertinents pour usage courant.
2. Ajouter un module de recherche internet séparé, basé sur SearXNG, capable de tourner avec plusieurs replicas en Kubernetes.

3. Documenter une architecture d’egress multi-régions cohérente sur Scaleway.
4. Mettre à jour la documentation et le prompt lui-même pour que ce besoin soit rejouable.

Amendement d’architecture attendu :

Le repository ne doit pas prétendre qu’un unique cluster Kubernetes Kapsule peut faire tourner nativement des pods répartis entre plusieurs régions Scaleway comme s’il s’agissait d’un seul plan de données global.

Si une exigence de sorties internet multi-régions est demandée, la solution propre doit être :

- cluster principal dans une région Kapsule
- pods SearXNG dans ce cluster
- trois nœuds ou pools de proxy d’egress séparés, hors du cluster ou gérés à part, chacun dans une région Scaleway différente
- distribution des requêtes SearXNG vers ces trois proxys

Le repository généré doit explicitement l’expliquer.

Autre amendement important :

Si le besoin métier dit "10 IPs de sortie par nœud", le repository ne doit pas l’implémenter naïvement sans discussion.

Il doit documenter la contrainte réelle du provider et proposer une solution plus réaliste si nécessaire, par exemple :

- 5 IPv4 flexibles + 5 IPv6 publiques sur un nœud si cela suffit
- ou 2 nœuds de proxy par région si 10 IPv4 distinctes sont réellement requises

MODULE DE RECHERCHE INTERNET

Le repository généré doit ajouter un module de recherche internet séparé basé sur https://github.com/searxng/searxng

Exigences minimales :

- profil Docker Compose local dédié, par exemple `search`, lançant au minimum `searxng` et `search-valkey`
- endpoint local simple de test, par exemple `http://localhost:8083/search`
- script de vérification locale permettant d’exécuter une requête JSON de test contre SearXNG

- déploiement Kubernetes distinct pour SearXNG
- service Kubernetes dédié
- ingress TLS dédié
- backend Valkey/Redis si nécessaire pour le limiter et la robustesse multi-replicas
- nombre de replicas paramétrable
- configuration d’un profil de moteurs de recherche généralistes qui privilégie les moteurs plus alignés avec des valeurs d’anonymat et de confidentialité

Le repository généré doit éviter une interprétation naïve de "tous les grands moteurs". Il doit privilégier par défaut :

- DuckDuckGo
- Brave
- Startpage
- Qwant
- Mojeek
- Wikipédia

et peut aussi exposer des moteurs plus grand public comme Bing et Google, mais avec des poids moindres que les moteurs plus alignés avec la confidentialité.

Si Google est activé, le repository doit documenter explicitement qu’il reste moins favorisé dans le ranking et plus exposé aux contre-mesures anti-bot que les moteurs plus respectueux de la confidentialité.

Pour l’expérience locale Docker, le repository peut utiliser un egress direct plus simple afin de rester facilement testable. Il doit documenter explicitement que :

- la topologie locale n’émule pas les trois proxys multi-régions

Exemples attendus :

- Kubernetes : `http://searxng/search`

La documentation doit préciser qu’en local, Open WebUI ne peut utiliser cette recherche que si le profil Docker `search` a effectivement été démarré.

Provider par défaut à configurer dans le repository généré :

- SCW_LLM_MODEL = mistral-small-3.2-24b-instruct-2506

Le repository généré peut conserver une compatibilité avec les anciens noms OPENAI_API_BASE, OPENAI_API_KEY et OPENAI_MODEL, mais SCW_LLM_BASE_URL, SCW_SECRET_KEY_LLM et SCW_LLM_MODEL doivent être la surface de configuration primaire partout :

- .env

- graphrag/settings.yaml
- scripts shell

Le choix embeddings doit être paramétrable par variable d’environnement.

Le fichier graphrag/settings.yaml doit utiliser :

- SCW_LLM_BASE_URL
- SCW_SECRET_KEY_LLM
- SCW_LLM_MODEL

Le repository généré ne doit pas supposer un ancien format GraphRAG.

Il doit aligner graphrag/settings.yaml sur la vraie version de la CLI installée dans l’image.

En pratique, avec une CLI GraphRAG récente, il faut utiliser un schéma de type :

- completion_models
- embedding_models
- input
- input_storage
- output_storage
- cache
- vector_store
- local_search
- global_search

et non un ancien schéma limité à :

- llm
- embeddings
- storage

Le cache GraphRAG ne doit pas être configuré avec cache.type=file si la version installée attend un cache logique de type json, memory ou none.

Pour une CLI récente, le repository généré doit utiliser une configuration compatible, par exemple :

- cache.type=json
- cache.storage.type=file
- input_storage.type=file
- output_storage.type=file

Le corpus de démonstration local étant en Markdown, la configuration d’entrée GraphRAG doit explicitement lire les fichiers .md, par exemple avec un file_pattern compatible Markdown et texte brut.

Le vector_store doit être compatible avec la dimension du modèle d’embeddings choisi.

Pour bge-multilingual-gemma2, le repository généré doit configurer la dimension 3584 si la version GraphRAG utilisée l’exige.

Le repository généré doit aussi tenir compte des particularités du provider embeddings réellement utilisé.

Par exemple, avec certaines combinaisons GraphRAG + LiteLLM + endpoint Scaleway, il peut être nécessaire de définir explicitement un call_args compatible pour les embeddings, comme encoding_format=float, afin d’éviter une erreur de validation côté provider.

Pour faciliter la visualisation des relations entre entités, le repository généré doit activer la production d’un snapshot GraphML si la version GraphRAG le supporte, par exemple via snapshots.graphml=true.

Si un visualiseur de graphe HTTP est fourni :

- il doit être servi par le bridge FastAPI
- il peut lire les artefacts GraphRAG réels comme entities.parquet, relationships.parquet, documents.parquet et graph.graphml
- il doit exposer au minimum une page HTML /graph, une API JSON /graph/data et éventuellement un téléchargement brut /graph/raw
- le pipeline Open WebUI peut ajouter un lien vers ce visualiseur dans la réponse finale, mais ne doit pas prétendre qu’Open WebUI visualise lui-même le graphe nativement
- il est préférable d’utiliser une bibliothèque web dédiée de type Cytoscape.js plutôt qu’un rendu Canvas maison difficile à maintenir

Si le repository choisit d’implémenter ce visualiseur :

- l’interface doit être en français
- l’ergonomie doit être soignée et institutionnelle, de type DSFR-like
- les boutons, formulaires, titres, états de chargement et panneaux de détails doivent être cohérents avec cette direction visuelle
- il est acceptable d’utiliser un thème maison inspiré du DSFR si les assets DSFR officiels ne sont pas intégrés
- si Cytoscape.js est utilisé, le README doit indiquer clairement si la librairie est servie localement ou chargée depuis un CDN avec fallback
- si une vue chronologique est proposée, elle doit fonctionner sur axe vertical et horizontal
- le guide temporel doit rester visible sur les deux axes et se recaler correctement après zoom, pan ou fit Cytoscape
- en orientation horizontale, le guide temporel ne doit pas être simplement collé au bas théorique du conteneur ; il doit rester positionné par rapport à la fenêtre visible du navigateur si une partie du viewer est hors écran
- le formulaire doit permettre de relire la question complète sous le champ "Votre question", sans dépendre de la largeur de l’input
- le centrage initial du graphe et le bouton de recentrage doivent utiliser la zone réellement visible du viewer dans la fenêtre, et non la seule hauteur totale du conteneur Cytoscape
- si une partie du viewer est hors écran au chargement, le calcul de pan/zoom doit tenir compte de la portion visible dans le viewport navigateur
- le champ "Axe chronologique" doit être visuellement grisé et désactivé tant que la vue chronologique n’est pas le mode effectivement actif
- si le visualiseur sert aussi à préparer une synthèse, l’objet de sélection principal doit être le fragment textuel pertinent issu des artefacts GraphRAG, pas seulement le nœud du graphe
- le panneau de détail peut proposer des cases à cocher ou actions équivalentes pour retenir des fragments liés à l’entité ou au lien sélectionné
- les éléments retenus pour la synthèse doivent être signalés visuellement dans le graphe sans réutiliser la couleur de remplissage déjà utilisée pour la légende des sources
- le visualiseur doit pouvoir maintenir un panier ou panneau "Synthèse en cours" listant les fragments retenus et leurs sources
- la rubrique export doit permettre au minimum de télécharger :
- les fragments retenus au format Markdown ou texte
- un prompt composite prêt à rejouer dans un autre LLM, construit uniquement à partir des fragments retenus et de la question de travail

Le bridge FastAPI doit exposer dans /config le modèle chat et le modèle embeddings configurés, sans jamais exposer la clé API brute.

Le bridge doit aussi éviter qu’une requête interactive Open WebUI reste bloquée trop longtemps.

Le repository généré doit donc prévoir un budget de temps borné pour les chemins coûteux, par exemple :

- un timeout spécifique pour `graphrag query`
- un timeout spécifique pour la synthèse LLM finale
- une réponse fallback déterministe si le budget restant est insuffisant

Le pipeline Open WebUI ne doit pas envoyer aveuglément dans GraphRAG tous les prompts techniques générés par Open WebUI lui-même.

Par exemple, si Open WebUI réutilise le pipeline pour générer des suggestions de follow-up, des titres ou d’autres prompts techniques de post-traitement, le repository généré doit savoir les détecter et les court-circuiter proprement, afin d’éviter :

- des timeouts inutiles
- une surcharge du bridge
- des erreurs du type `timed out` ou `Remote end closed connection without response`

Quand le bridge renvoie une réponse issue de graphrag query, il doit aussi joindre des citations pertinentes pour la question posée. Il ne doit pas renvoyer simplement les premiers documents d’un manifeste générique si ces citations ne sont pas liées à la requête.

Le bridge doit aussi exposer un état permettant de distinguer :

- un index fallback simplement présent
- un vrai index GraphRAG prêt pour les requêtes CLI

Un booléen ou indicateur de type graphrag_query_ready est acceptable si nécessaire.

Cet indicateur ne doit pas devenir true trop tôt pendant l’indexation. Il doit refléter un état réellement interrogeable par graphrag query, et non la simple apparition de quelques artefacts intermédiaires.

Si SCW_SECRET_KEY_LLM n’est pas renseignée :

- le bridge doit continuer à fonctionner en mode fallback déterministe local
- les smoke tests Docker locaux doivent rester passants
- le README doit expliquer clairement cette différence entre mode fallback et mode LLM réel

Le README doit expliquer comment utiliser :

- Scaleway Generative APIs
- OpenAI
- Azure OpenAI
- vLLM compatible OpenAI
- endpoint interne compatible OpenAI

MODE 1 — LOCAL DOCKER

docker compose up -d

Dans ce mode :

- Open WebUI local

En local, si le repository expose un login OIDC Keycloak :

- Keycloak doit être accessible depuis le Mac sur http://localhost:8082
- le bouton OIDC d’Open WebUI ne doit pas rediriger le navigateur vers un hostname Docker interne de type keycloak:8080
- si le formulaire natif Open WebUI est masqué, le repository doit le faire explicitement avec ENABLE_LOGIN_FORM=false
- les utilisateurs Keycloak de test doivent pouvoir créer leur compte Open WebUI via OIDC local, donc ENABLE_OAUTH_SIGNUP=true doit être activé pour ce scénario
- le README doit expliquer que docker compose --profile sso up -d keycloak est nécessaire avant tout test OIDC local
- le README doit proposer le raccourci direct http://localhost:3000/oauth/oidc/login pour démarrer la connexion
- si le visualiseur GraphRAG est protégé par OIDC, il doit réutiliser Keycloak local et rediriger le navigateur vers http://localhost:8082 en cas de session absente ou expirée

Le nom de service Docker keycloak est interne au réseau Docker et n’est pas directement accessible depuis le navigateur du poste hôte.

La configuration locale doit donc produire des URLs navigateur de type localhost:8082 et non keycloak:8080.

En pratique, le repository généré doit gérer correctement ce point, par exemple avec :

- une metadata OIDC consommée par Open WebUI qui reste compatible avec le poste hôte
- une configuration Keycloak locale qui publie un issuer et un authorization_endpoint basés sur http://localhost:8082
- un realm local de test qui n’impose pas HTTPS pour ce scénario de développement
- si un client OIDC dédié au visualiseur est utilisé, ses redirect URIs et web origins locales doivent viser http://localhost:8081/* et http://127.0.0.1:8081/*

Le bootstrap initial d’Open WebUI peut prendre plusieurs minutes.

Les healthchecks et scripts de tests doivent attendre proprement :

- /healthz pour le bridge
- /health pour Open WebUI
- disponibilité du service pipelines

Les scripts shell doivent charger automatiquement .env quand c’est nécessaire.

Les fichiers .env et .env.example doivent être compatibles à la fois avec Docker Compose et avec source .env dans bash/zsh.

Ils ne doivent pas contenir plusieurs définitions contradictoires de la même variable.

Si des variables sont déjà présentes dans l’environnement du shell, par exemple via printenv :

- SCW_SECRET_KEY_LLM
- SCW_LLM_MODEL
- SCW_LLM_BASE_URL

alors les scripts et le déploiement ne doivent PAS écraser ces valeurs avec des placeholders provenant de .env.

La règle attendue est :

- les variables exportées dans l’environnement courant priment
- .env et .env.example servent de fallback
- un placeholder CHANGE_ME dans .env ne doit jamais remplacer une vraie valeur déjà exportée

Le repository généré doit fournir un chargement d’environnement cohérent et réutilisable dans les scripts shell pour garantir cette priorité.

Le repository généré doit éviter les pièges de ce type :

- une première occurrence ENABLE_OAUTH_SIGNUP=true
- puis plus bas une seconde occurrence ENABLE_OAUTH_SIGNUP=false

Dans ce cas, la dernière valeur gagne et peut casser silencieusement le login OIDC local.

Toute valeur contenant des espaces doit être quotée.

Exemple attendu :

GRAPHRAG_RESPONSE_TYPE="Multiple Paragraphs"

Le docker-compose local doit utiliser des bind mounts permettant à l’utilisateur d’accéder aux données depuis son poste :

- ./graphrag:/app/graphrag
- ./openwebui/data:/app/backend/data

En local Docker, le cache GraphRAG ne doit pas polluer le working tree du repository si une meilleure option existe.

Le repository généré doit préférer un volume Docker nommé dédié pour le cache GraphRAG, par exemple :

- volume local monté sur /app/graphrag/cache

Le README doit expliquer explicitement que ce cache local est distinct du corpus source versionné.

Le service bridge du docker-compose local doit aussi propager explicitement les variables SCW_LLM_BASE_URL, SCW_SECRET_KEY_LLM, SCW_LLM_MODEL et OPENAI_EMBEDDING_MODEL pour que des variables exportées dans le shell du poste soient réellement visibles dans le conteneur.

Le bridge ne doit pas considérer un simple index-manifest.json de fallback comme preuve qu’un vrai index GraphRAG est prêt pour les requêtes CLI.

Pour tenter une requête GraphRAG CLI, le repository généré doit vérifier la présence d’artefacts réels et finaux, par exemple :

- entities.parquet
- relationships.parquet
- communities.parquet
- community_reports.parquet
- fichier graph.graphml
- dossier lancedb contenant au moins une table .lance

Le README doit expliquer que l’utilisateur peut modifier graphrag/input directement depuis sa machine puis relancer ./scripts/index_corpus.sh.

Les certificats TLS doivent être générés automatiquement avec Let's Encrypt via cert-manager.

Si Keycloak et Open WebUI sont publiés derrière ingress-nginx, le repository généré doit anticiper la taille des en-têtes de réponse OIDC, notamment les cookies de session et certaines redirections avec `redirect_uri` longue.

Au minimum, les ingress concernés doivent prévoir un buffer de proxy suffisant, par exemple :

- `nginx.ingress.kubernetes.io/proxy-buffer-size: "32k"`
- et, si nécessaire, un réglage cohérent de `proxy-buffers-number`

Sinon un login OIDC valide peut échouer avec un `502` sur l’authorization endpoint Keycloak via l’ingress, avec un log NGINX du type :

- `upstream sent too big header while reading response header from upstream`

Si la version GraphRAG embarquée ne propose pas nativement un backend de cache S3, le repository ne doit pas prétendre le supporter directement.

Dans ce cas, la solution attendue en Kubernetes est pragmatique :

- cache GraphRAG restant de type fichier dans le pod
- répertoire de cache monté localement dans le pod, par exemple sur /data/graphrag/cache
- synchronisation pull depuis S3 avant indexation ou démarrage
- synchronisation push vers S3 après indexation et lors d’un arrêt propre si pertinent

Le repository généré doit alors prévoir les variables d’environnement correspondantes, par exemple :

- GRAPHRAG_CACHE_S3_ENABLED
- GRAPHRAG_CACHE_S3_BUCKET
- GRAPHRAG_CACHE_S3_PREFIX
- GRAPHRAG_CACHE_S3_REGION
- GRAPHRAG_CACHE_S3_ENDPOINT_URL
- GRAPHRAG_CACHE_S3_ACCESS_KEY_ID
- GRAPHRAG_CACHE_S3_SECRET_ACCESS_KEY
- GRAPHRAG_CACHE_S3_SESSION_TOKEN

Les manifests Kubernetes doivent refléter cette stratégie, par exemple avec :

- emptyDir pour le cache local du pod
- init container de pull S3
- job d’indexation qui pull puis push le cache
- hook preStop ou mécanisme équivalent pour republier le cache si nécessaire

Les tests locaux Docker doivent réellement être exécutables et vérifier au minimum :

- GET http://localhost:8081/healthz retourne 200
- POST http://localhost:8081/index retourne 200
- POST http://localhost:8081/query retourne une réponse non vide
- GET http://localhost:3000/health retourne un JSON avec status=true
- le service pipelines répond sur son API

Si le profil SSO local est lancé, les tests doivent aussi vérifier :

- GET http://localhost:3000/oauth/oidc/login retourne une redirection 302
- la redirection pointe vers une URL navigateur accessible depuis le poste hôte, typiquement http://localhost:8082/...
- la redirection ne doit pas pointer vers http://keycloak:8080/...
- le document de discovery local de Keycloak annonce un issuer et des endpoints en http://localhost:8082/...
- si le repository choisit un mode Keycloak uniquement, /api/config d’Open WebUI reflète enable_login_form=false et un provider oauth Keycloak
- le login OIDC d’un utilisateur de test comme user1@test.local aboutit réellement à une session Open WebUI, sans erreur 403 liée à ENABLE_OAUTH_SIGNUP
- les logs Open WebUI ne doivent pas contenir un rejet OAuth du type Access Prohibited après un callback OIDC valide
- si un visualiseur GraphRAG protégé est fourni, GET /graph/config indique clairement le mode auth attendu
- si un visualiseur GraphRAG protégé est fourni, GET /graph/data sans bearer token retourne 401
- si un client Keycloak graphrag-viewer est configuré, un vrai access token obtenu depuis Keycloak permet d’appeler /graph/data et /graph/raw avec succès
- la page /graph doit charger une configuration cohérente avec Keycloak local et déclencher un login-required quand la session navigateur est absente

En Kubernetes, si le repository publie Open WebUI et Keycloak derrière ingress-nginx, la validation doit aussi vérifier :

- qu’un appel au moins vers l’authorization endpoint Keycloak public ne retourne pas `502`
- qu’un flux OIDC avec `redirect_uri` longue reste fonctionnel pour un client comme `graphrag-viewer`
- que les logs récents du contrôleur ingress-nginx ne contiennent pas `upstream sent too big header while reading response header from upstream`
- que les manifests d’ingress Keycloak et Open WebUI configurent un `proxy-buffer-size` suffisant, par exemple `32k`

Le repository généré doit aussi proposer un test bout en bout dédié au corpus de démonstration qui :

- génère le corpus Wikipédia
- lance l’indexation
- exécute une requête représentative
- vérifie que la réponse est non vide
- vérifie que des citations sont retournées
- si un visualiseur est implémenté, vérifie aussi que la réponse du bridge ou du pipeline contient un lien de graphe exploitable

Si SCW_SECRET_KEY_LLM n’est pas fournie, ce test doit rester valide en mode fallback et documenter explicitement les moteurs observés, par exemple :

- POST /index peut répondre avec engine_used=fallback-manifest
- POST /query peut répondre avec engine_used=local-deterministic
- des citations non vides doivent quand même être présentes

Si des variables SCW_* réelles sont présentes dans l’environnement du shell, la validation locale doit aussi vérifier que le conteneur bridge les consomme effectivement.

Par exemple, la validation peut confirmer que :

- le conteneur bridge voit SCW_LLM_BASE_URL
- le conteneur bridge voit SCW_LLM_MODEL
- la clé n’est jamais exposée brute, mais /config indique openai_api_key_configured=true
- le modèle effectif visible dans /config correspond bien à la valeur injectée depuis l’environnement réel

Si SCW_SECRET_KEY_LLM est fournie et que le provider répond réellement, la validation ne doit pas se contenter d’un fallback fonctionnel.

Elle doit aussi vérifier que la vraie CLI GraphRAG fonctionne avec la version embarquée :

- POST /index retourne engine_used=graphrag-cli au moins une fois
- le dossier graphrag/output contient de vrais artefacts GraphRAG
- la présence du seul index-manifest.json n’est pas suffisante pour déclarer GraphRAG opérationnel
- une requête CLI GraphRAG réelle fonctionne avec la syntaxe attendue par la version installée

Le repository généré doit aussi tenir compte de l’évolution de la syntaxe CLI.

Par exemple, une CLI GraphRAG récente attend :

- graphrag query --root <dir> --method <local|global> "question"

et non :

- graphrag query --query "question"

La validation ne doit pas s’arrêter au fait que le module pipeline est chargé.

Elle doit vérifier que le pipeline est réellement consommable par Open WebUI.

Concrètement, le repository généré doit satisfaire les critères suivants :

- GET /v1/pipelines du service pipelines retourne au moins un pipeline GraphRAG
- GET /v1/models du service pipelines retourne une liste non vide exploitable par Open WebUI
- POST /v1/chat/completions du service pipelines retourne 200 avec un modèle GraphRAG valide
- Open WebUI peut appeler ce modèle sans erreur 404

Si nécessaire, adapter le code du pipeline ou la structure attendue par l’image ghcr.io/open-webui/pipelines pour que ces critères soient réellement respectés.

RETOURS D’EXPLOITATION A INTEGRER DES LE DEPART

Le repository généré doit intégrer explicitement les leçons suivantes au lieu de les découvrir plus tard en exploitation :

- le déploiement Kubernetes ne doit pas exiger une indexation GraphRAG bloquante pour être considéré comme réussi
- `deploy/deploy-k8s.sh` doit déployer la stack et rendre `mygraph` utilisable sans attendre `job/graphrag-index`
- le job d’indexation GraphRAG doit rester disponible mais devenir opt-in, par exemple via `RUN_GRAPHRAG_INDEX_JOB=true`
- la documentation doit expliquer que la visualisation de secours est un graphe documentaire de corpus, tandis que le graphe GraphRAG complet reste optionnel
- quand l’indexation complète est activée sur Scaleway Kubernetes, il faut prévoir un timeout long, de l’ordre de `3600s`, car un corpus réel peut dépasser largement les timeouts trop courts
- l’image `bridge` doit être buildée explicitement en `linux/amd64` pour le cluster Scaleway ciblé si les nœuds ne sont pas homogènes avec la machine de build
- le déploiement Keycloak derrière ingress doit utiliser le bon hostname public et des headers proxy fiables ; en pratique il faut prévoir `--hostname=<host public>` et `--proxy-headers=xforwarded`
- la configuration OIDC ne doit jamais produire des redirections du type `http://https//...` ; le repository doit vérifier ce flux en test
- le module SearXNG doit refléter les contraintes réelles d’exploitation observées : DuckDuckGo peut être trop instable à cause des CAPTCHA et doit pouvoir être retiré proprement du profil par défaut
- pour des appels backend Open WebUI -> SearXNG en HTTP interne, le limiter SearXNG peut renvoyer des `429` même quand le service semble sain ; le repository doit soit désactiver ce limiter pour le trafic interne, soit séparer un endpoint interne non limité d’un endpoint public protégé
- le repository doit inclure un test Kubernetes de recherche web qui regarde aussi les logs SearXNG et résume les refus moteur (`CAPTCHA`, `403`, `429`, `blocked`, `refused`)
- la rotation des mots de passe Keycloak du realm ne doit jamais écrire les secrets rotatés dans les fichiers versionnés ; il faut un store local ignoré par Git, par exemple `keycloak/realm-passwords.local.json`
- le rendu des manifests ne doit pas faire un `envsubst` global sans garde-fou ; il faut limiter explicitement la liste des variables autorisées pour éviter des substitutions accidentelles
- la documentation doit expliquer que les URL temporaires Keycloak `login-actions/authenticate?...session_code=...` sont jetables et qu’une erreur `invalid_code` après redémarrage n’implique pas nécessairement un bug persistant
